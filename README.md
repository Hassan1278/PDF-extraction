# PDF Extraction Pipeline

A schema-constrained document extraction pipeline that accepts a PDF and a JSON Schema and returns structured, validated JSON — powered by Groq's API (Llama 4 Scout).

## What it does

1. Inspects each PDF page using spatial bounding-box coverage to classify it as `text`, `image`, `mixed`, or `empty`
2. Extracts text digitally (pdfplumber) or via OCR (easyocr) depending on page type
3. Builds a structured prompt and sends it to the LLM with a `response_format` constraint — the model is forced to output valid JSON matching your schema, no post-hoc parsing
4. Validates the result against the schema and retries on failure
5. Returns the extracted data along with debug metadata

## Stack

| Purpose | Library |
|---|---|
| PDF text extraction | pdfplumber |
| PDF rendering | PyMuPDF (fitz) |
| OCR | easyocr |
| LLM inference | Groq API (Llama 4 Scout) |
| Schema validation | jsonschema + pydantic |
| API | FastAPI |
| CLI | Typer |
| Tests | pytest |
| Linting | ruff |

> **Why Groq?** Running a local LLM requires significant GPU resources. Groq provides hardware-accelerated inference via API, giving comparable speed and quality without needing local hardware. The client is straightforward to swap — update the provider and model name in `groq_client.py` to point at any OpenAI-compatible endpoint (e.g. a self-hosted vLLM server).

### Python

```python
from src.doc_extract.pipeline import run_pipeline
from pathlib import Path
import json

schema = json.loads(Path("schemas/default_schema.json").read_text())
result = run_pipeline(Path("document.pdf"), schema)

print(result.valid)   # True / False
print(result.data)    # extracted fields
print(result.errors)  # validation errors if any
```

## Schemas

Schemas are standard JSON Schema objects. Example:

```json
{
  "type": "object",
  "properties": {
    "sender":  { "type": "string", "description": "Name of the sender" },
    "date":    { "type": ["string", "null"], "description": "Date in DD.MM.YYYY" },
    "summary": { "type": "string", "description": "Brief summary of the document" }
  },
  "required": ["sender", "date"]
}
```

Fields described as "summary" are generated by the model. All other fields are extracted from the document.

Bundled schemas: `default_schema.json`, `invoice_schema.json`, `id_form_schema.json`, `contract_schema.json`.

## How page routing works

Each page is classified by measuring what fraction of the page area is covered by word bounding boxes (text coverage) and image bounding boxes (image coverage):

| Classification | Condition |
|---|---|
| `text` | text coverage > 20%, image coverage ≤ 15% |
| `image` | image coverage > 15%, text coverage ≤ 20% |
| `mixed` | both above threshold |
| `empty` | neither above threshold |

A char-count fallback applies: if coverage is below threshold but pdfplumber extracted ≥ 50 characters, the page is reclassified as `text`. This handles sparse layouts (short invoices, forms) where word bounding boxes cover little page area despite real content being present.

For `image` and `mixed` pages, the pipeline first attempts to extract digital text via pdfplumber. If the result is fewer than 100 characters, it falls back to OCR.

## Output

```python
ExtractionResult(
    request_id="...",     # UUID for tracing
    pages=2,              # total pages processed
    valid=True,           # schema validation passed
    data={...},           # extracted fields
    errors=[],            # validation errors or page failures
    retry_count=0,        # total retries across all pages
    pages_succeeded=[0, 1],
    pages_failed=[]
)
```

## Evaluation

Run the evaluation harness against the labeled test set:

```bash
python -m src.doc_extract.eval.runner
```

The eval set (`eval/ground_truth.json`) contains 40 labeled documents across 3 document types and 7 layout variants. Sample PDFs are not committed — regenerate them with:

```bash
python scripts/generate_sample_pdfs.py
```

### Results (n=40)

| Metric | Result |
|---|---|
| Schema valid rate | 100% (40/40) |
| Field exact match | 100% |
| Avg latency | 0.87s / document |
| Hard failures | 0 |

| Layout type | Cases | Pass |
|---|---|---|
| Standard labeled fields | 30 | 30/30 |
| Prose-embedded data | 2 | 2/2 |
| Ambiguous labels ("Issued by", "Billed to") | 1 | 1/1 |
| Reversed layout (recipient before sender) | 1 | 1/1 |
| Noisy (3 dates on page, model picks correct one) | 1 | 1/1 |
| Unusual field labels ("DoB", "Document No.") | 1 | 1/1 |
| Split name fields (first + last separate) | 1 | 1/1 |
| Multi-page contract | 1 | 1/1 |
| Footer-only sender | 1 | 1/1 |

Generative fields (e.g. `summary`) are excluded from exact-match scoring.

**Known limitation:** eval set consists of clean, born-digital PDFs. Scanned documents would stress-test the OCR path, which these results do not cover.
